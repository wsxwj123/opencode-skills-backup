## 一、状态管理器（state_manager.py）

借鉴 sci2doc 双层设计，简化为申请书体量：

### 1.1 核心命令

```python
state_manager.py init            # 初始化项目，创建目录结构
state_manager.py profile         # 配置申请书参数（含与用户协商字数）
state_manager.py load            # 加载状态
                  --section P1   # 仅加载指定节 + 关联consistency_map
                  --minimal      # 最小化加载（仅key_facts）
                  --global       # 全局加载（自审/合并时用）
state_manager.py update          # 更新状态（默认merge模式）
state_manager.py snapshot        # 创建快照（含所有sections + data/）
state_manager.py rollback        # 回滚到指定快照
state_manager.py word-count      # 中文字数统计（按节/全文）
state_manager.py page-estimate   # 页数估算（按每页~800字）
state_manager.py write-cycle     # 预写门控：加载上下文 + token预算分配
state_manager.py self-review     # 触发自审模块
state_manager.py sync-all        # 强制同步所有历史文件（防遗漏）
```

### 1.2 关键设计决策

- `load --section P1 --minimal`：只加载当前节 + consistency_map中与该节关联的摘要，不加载全文
- `load --global`：加载全局上下文（仅在跨节一致性检查和最终自审时使用）
- Token预算默认4000（申请书单节体量远小于论文章节）
- 每次 `write-cycle` 自动注入：当前节已有内容摘要 + 关联节key_facts + consistency_map相关条目

### 1.3 sync-all 强制同步机制

每个小节撰写/修改完成后，`sync-all` 自动检查并更新：

| 文件 | 检查内容 |
|------|---------|
| `consistency_map.json` | 新增/修改的映射条目是否已注册 |
| `literature_index.json` | 新引用的文献是否已录入+验证状态 |
| `context_memory.md` | 本次操作摘要是否已追加 |
| `project_state.json` | 当前Phase和门控状态是否已更新 |
| `history_log.json` | 操作记录是否已追加 |

**任何一项未同步则阻断进入下一Phase。**

## 二、跨节一致性映射表（consistency_map.json）

### 2.1 四维一一对应模型

这是本技能最核心的逻辑约束。假说(H)、目标(O)、研究内容(RC)、关键科学问题(KSQ)必须严格一一对应：

```
H-1  ←→  O-1  ←→  RC-1  ←→  KSQ-1
H-2  ←→  O-2  ←→  RC-2  ←→  KSQ-2
H-3  ←→  O-3  ←→  RC-3  ←→  KSQ-3
```

**研究目标 vs 关键科学问题（必须区分）：**

| | 研究目标（Objective） | 关键科学问题（Key Scientific Problem） |
|---|---|---|
| 本质 | 你要**做什么** | 你要**回答什么** |
| 表述 | 行动导向："阐明XXX的机制" | 问题导向："XXX如何调控YYY？" |
| 粒度 | 可操作、可考核的工作目标 | 科学层面的未知/待解答的问题 |
| 来源 | 从研究假说推导而来 | 从科学问题(SQ)凝练而来 |

### 2.3 自动验证规则

| 编号 | 规则 | 严重级别 |
|------|------|---------|
| V-01 | 每个 SQ 必须至少映射到一个 H 和一个 KSQ | ERROR |
| V-02 | H / O / RC / KSQ 必须一一对应（数量相等，映射完整） | ERROR |
| V-03 | 每个 RC 必须至少映射到一个 M（方法） | ERROR |
| V-04 | 每个 RC 必须关联 annual_plan_year | WARNING |
| V-05 | 每个 IN 必须追溯到具体的 RC 和 M | ERROR |
| V-06 | 每个 M 必须有至少一个 F（可行性证据） | ERROR |
| V-07 | F 来源必须为 P3_1 或 P3_2 | WARNING |
| V-08 | 关键词必须在摘要、P1、P2中均出现 | WARNING |
| V-09 | M 的 budget_trace 应能追溯到 budget_table.json | INFO |
| V-10 | 无孤立条目（任何实体无映射则触发警告） | WARNING |

## 三、反遗忘机制

### 三层防线

| 层级 | 机制 | 触发时机 | 内容 |
|------|------|---------|------|
| L1 | 节级摘要 | 每节完成后自动提取 | key_facts ≤10条，每条≤80字 |
| L2 | consistency_map | 写新节时自动注入关联条目 | 仅注入与当前节相关的映射 |
| L3 | context_memory.md | 每次交互结束追加 | 时间戳+操作摘要+关键决策+用户确认 |

**与sci2doc的区别：** sci2doc需要chapter_section_digests因为论文8万字+。申请书~2万字，L1+L2+L3足够，不需要额外digest。

### 强制同步规则

每个小节撰写/修改时，以下文件**必须**同步更新：

```
[撰写/修改任意section]
    │
    ├─→ consistency_map.json    更新映射条目
    ├─→ literature_index.json   更新文献索引（如有新引用）
    ├─→ context_memory.md       追加操作摘要
    ├─→ project_state.json      更新Phase/门控状态
    └─→ history_log.json        追加操作记录
```

**违反同步规则 → 阻断进入下一Phase。**

## 四、Token预算管理

```json
{
  "token_budget": {
    "default_per_section": 4000,
    "global_load_max": 8000,
    "consistency_map_max": 1500,
    "literature_context_max": 2000,
    "tail_trim_threshold": 0.85,
    "compact_mode_threshold": 0.95
  }
}
```

### 预算分配策略

**节级写作：**
- 当前节已有内容摘要：40%
- consistency_map相关条目：20%
- 文献上下文（仅P1需要）：25%
- 系统指令：15%

**全局自审：**
- 全文摘要：50%
- consistency_map完整：30%
- 系统指令：20%

**超预算降级：**
1. 先裁剪文献上下文
2. 再压缩consistency_map为仅ID+标题
3. 最后启用compact模式（仅保留key_facts）
