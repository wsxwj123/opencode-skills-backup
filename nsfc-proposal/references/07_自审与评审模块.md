# 07 自审与评审模块

---

## 一、模块定位

自审模块和评审模块**合并为统一评审体系**，分两个层级：

| 层级 | 名称 | 触发时机 | 范围 |
|------|------|---------|------|
| L1 | 节级自审（Quick Review） | 每个小节撰写/润色完成后 | 单节 |
| L2 | 全文评审（Full Review） | Phase 7 / Polish Step 5 | 全文 |

两个层级使用**同一套评审维度**，但检查深度不同。

---

## 二、评审维度体系

### 2.1 完整维度表（10维度）

```yaml
dimensions:
  # ═══ 学术质量（4维）═══
  D-01_科学意义与立项依据:
    description: "研究问题是否具有重要科学价值，立项依据是否充分、逻辑是否严密"
    applicable_sections: ["P1"]
    L1_check: "逻辑链完整性、科学问题是否清晰凝练"
    L2_check: "SQ是否有充分文献支撑、是否有真正的科学空白而非伪问题"

  D-02_创新性:
    description: "是否有明确创新点，创新点是否有实质内容而非空话"
    applicable_sections: ["P2"]
    L1_check: "IN条目是否可追溯到RC和M"
    L2_check: "创新点是否真正新颖、是否与现有文献区分、是否避免了夸大表述"

  D-03_研究方案合理性:
    description: "技术路线是否合理、实验设计是否科学、关键技术是否可行"
    applicable_sections: ["P2"]
    L1_check: "M条目是否完整、是否覆盖所有RC"
    L2_check: "方案是否有备选路线、关键技术难点是否有应对、时间安排是否合理"

  D-04_可行性:
    description: "前期基础是否支撑研究方案，工作条件是否满足需求"
    applicable_sections: ["P3_1", "P3_2"]
    L1_check: "每个M是否有F支撑"
    L2_check: "F的质量是否足够（发表论文的相关性、设备的匹配度）"

  # ═══ 逻辑一致性（2维）═══
  D-05_四维对应完整性:
    description: "假说-目标-内容-问题是否一一对应、无遗漏无冲突"
    applicable_sections: ["P2", "全局"]
    L1_check: "当前节涉及的映射是否完整"
    L2_check: "consistency_map V-01~V-10全部验证"

  D-06_跨节逻辑一致性:
    description: "各节之间术语、表述、逻辑是否自洽"
    applicable_sections: ["全局"]
    L1_check: "不适用（仅L2）"
    L2_check: "关键词一致性、SQ→H→O→RC→KSQ→M→IN→F全链路追踪"

  # ═══ 写作质量（2维）═══
  D-07_写作风格:
    description: "是否段落式叙事、无AI痕迹、无空话套话、无禁用句式/修辞"
    applicable_sections: ["全部"]
    L1_check: "禁用词/句式/修辞扫描、列点检测"
    L2_check: "节奏控制、段落开头多样性、全文语气一致性"

  D-08_格式规范:
    description: "是否符合NSFC格式要求，页数是否达标，三线表是否合规"
    applicable_sections: ["全部"]
    L1_check: "字数是否在协商范围内"
    L2_check: "总页数≤30、三线表格式、AI声明、参考文献格式"

  # ═══ 预算（1维）═══
  D-09_预算合理性:
    description: "经费预算是否与研究内容匹配，测算是否合理"
    applicable_sections: ["B1", "B2", "B3"]
    L1_check: "预算项是否可追溯到M"
    L2_check: "单价是否合理、总额是否匹配、分类是否正确"

  # ═══ 摘要（1维）═══
  D-10_摘要质量:
    description: "是否精准概括全文核心，字数达标，关键词恰当"
    applicable_sections: ["00_摘要"]
    L1_check: "字数≤400、关键词与consistency_map一致"
    L2_check: "是否覆盖研究意义/目标/内容/方法/预期成果"
```

### 2.2 评分体系

每个维度评分：

| 等级 | 含义 | 行动 |
|------|------|------|
| A | 优秀，无需修改 | 通过 |
| B | 良好，有小问题 | 建议修改，不阻断 |
| C | 存在明显问题 | 必须修改 |
| D | 严重缺陷 | 必须重写该部分 |

**门控规则：**
- L1自审：任何维度为D → 阻断，必须修正后重新自审
- L2全文评审：任何维度为D → 阻断；超过3个维度为C → 建议整体修订

## 三、L1 节级自审（Quick Review）

### 3.1 触发时机

每个小节撰写/润色完成后，HALT之前自动执行。

### 3.2 执行流程

```
1. 加载当前节内容
2. 加载 consistency_map 中与当前节相关的条目
3. 根据 applicable_sections 筛选适用维度
4. 对每个适用维度执行 L1_check
5. 生成节级评审报告
6. 如有D级 → 自动修正 → 重新自审
7. 报告随HALT一起展示给用户
```

### 3.3 输出格式

```
═══ 节级自审报告：P2_2.2.1_纳米药物的构建与表征 ═══

【D-03 研究方案合理性】评分：B
  诊断：M-3（动态光散射表征）缺少具体参数设置描述
  锚点：第3段第2句
  建议：补充DLS测量条件（温度、角度、溶剂）

【D-07 写作风格】评分：C
  诊断：第2段使用了"不仅...而且..."句式；第5段使用项目符号列表
  锚点：第2段第4句、第5段
  建议：拆分递进句为独立陈述；将列表改为段落式阐述

【D-05 四维对应】评分：A
  诊断：RC-1的方法M-1、M-2均已描述，映射完整

总评：B（需修正D-07后通过）
```

## 四、L2 全文评审（Full Review）

### 4.1 触发时机

- Write Mode Phase 7
- Polish Mode Step 5
- 用户手动触发 `state_manager.py self-review`

### 4.2 执行流程

```
1. load --global（加载全文摘要 + 完整consistency_map）
2. 执行全部10个维度的L2_check
3. 执行 consistency_map 验证规则 V-01~V-10
4. 执行引用-索引一致性矩阵检查
5. 执行页数估算
6. 生成全文评审报告
7. HALT → 用户审阅
```

## 六、Polish Mode专用：严格评审报告（参照sci2doc规则）

### 6.1 与L2全文评审的区别

L2全文评审是Write/Polish共用的标准化评审流程。Polish Mode在Step 1额外生成一份**严格评审报告**（`data/polish_review_report.md`），其特点：

- 以真实基金评审专家的视角撰写，段落式叙事
- 不仅检查格式和一致性，更审视研究设计的科学合理性
- 对假说的可证伪性、实验设计的严谨性、创新点的实质性做出判断
- 参照sci2doc skill的评审规则，执行结构化的forensic检查
- 输出为人类可读的完整评审意见书，写入本地文件

### 6.2 sci2doc式评审检查清单

在标准10维度之外，Polish Mode评审额外执行：

```yaml
forensic_checks:
  FC-01_假说可证伪性:
    check: "每个假说是否可以通过实验被证伪？"
    fail_example: "'纳米药物具有良好的生物相容性'——这是预期结果，不是假说"
    pass_example: "'X修饰可将溶血率降低至5%以下'——可通过实验验证或否定"

  FC-02_对照组完整性:
    check: "每个实验是否设置了充分的对照组？"
    required: "阳性对照、阴性对照、空白对照（视实验类型）"

  FC-03_统计方法预设:
    check: "是否预设了统计分析方法和样本量依据？"

  FC-04_逻辑链完整性:
    check: "SQ→H→O→RC→KSQ→M→IN→F 全链路是否无断裂？"

  FC-05_创新点实质性:
    check: "创新点是否有具体的技术/方法/理论突破，而非空泛表述？"
    fail_example: "'首次系统研究了...'——'首次'和'系统'都是空话"
    pass_example: "'利用X策略实现了Y效果，较现有方法Z提升了N倍'"

  FC-06_文献覆盖度:
    check: "是否遗漏了领域内的关键文献？近2年进展是否覆盖？"

  FC-07_预算-方法匹配:
    check: "预算分配是否与研究方法的实际需求匹配？"
```

### 6.3 评审报告生成后的工作流

```
1. 评审报告写入 data/polish_review_report.md
2. 结构化数据写入 data/diagnosis_report.json
3. HALT → 展示报告 → 用户审阅
4. 用户逐条确认/反驳评审意见
5. 形成修改计划 → 写入 proposal_profile.json 的 polish_plan
6. 按修改计划逐条执行（见06_Polish_Mode流程.md Step 3）
```
