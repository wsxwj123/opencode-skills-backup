# 04 文献管理

---

## 一、检索工具约束（强制）

### 1.1 允许使用的工具

| 优先级 | 工具 | 用途 |
|--------|------|------|
| 1（首选） | `paper-search_search_pubmed` | 生物医学领域主检索 |
| 2（补充） | `paper-search_search_semantic` | 跨学科补充、引用网络追踪 |
| 3（扩展） | `paper-search_search_google_scholar` | 扩展检索、查漏补缺 |
| 4（预印本） | `paper-search_search_biorxiv` / `paper-search_search_medrxiv` | 最新未发表研究 |
| 5（全文获取） | `paper-search_read_arxiv_paper` / `paper-search_read_semantic_paper` | 需要阅读全文时 |

### 1.2 禁止使用的工具

以下工具**绝对禁止**用于学术文献检索：

- `tavily_tavily-search` — 通用网页搜索，非学术数据库
- `websearch_web_search_exa` — 通用网页搜索
- `webfetch` — 网页抓取
- `fetch_fetch` — 通用URL抓取

**原因：** 通用搜索引擎无法保证文献的真实性、完整性和学术规范性。所有文献必须来自学术数据库。

## 二、检索策略（两轮模型）

### Round 1：立项依据阶段（Phase 1）

**目标：** 建立完整文献库，≥30篇核心文献

```
Step 1: PubMed主检索
  - 用3-5组关键词组合检索
  - 每组取top 10-20结果
  - 筛选：相关性 + 影响因子 + 发表年份

Step 2: Semantic Scholar补充
  - 用PubMed结果中的核心论文做引用网络追踪
  - 找到PubMed可能遗漏的跨学科文献

Step 3: Google Scholar扩展
  - 查漏补缺，特别是中文期刊论文
  - 确保中文期刊≥5篇

Step 4: 文献筛选与分类
  - 按role分类：gap_evidence / method_support / prior_work / comparison
  - 确认满足引文指标
```

### Round 2：研究内容阶段（Phase 2）

**目标：** 补充方法学文献和最新进展

```
- 针对技术路线中的关键方法，检索方法学原始文献
- 检索最近6个月的最新进展（bioRxiv/medRxiv）
- 更新literature_index.json
```

### 引文指标门控

```json
{
  "citation_targets": {
    "min_total": 30,
    "min_recent_5yr": 20,
    "min_cn_journals": 5
  }
}
```

Phase 1完成后自动检查，不满足则补充检索。

## 三、文献索引结构（literature_index.json）

```json
{
  "metadata": {
    "total_count": 0,
    "recent_5yr_count": 0,
    "cn_journal_count": 0,
    "last_updated": "",
    "verification_status": "pending"
  },
  "entries": [
    {
      "id": "L-001",
      "ref_number": 1,
      "authors": "Zhang Y, Li X, Wang H, et al.",
      "year": 2024,
      "title": "完整论文标题",
      "journal": "Nature Communications",
      "volume": "15",
      "pages": "1234",
      "doi": "10.1038/s41467-024-xxxxx",
      "pmid": "12345678",
      "is_cn_journal": false,
      "is_recent_5yr": true,
      "used_in_sections": ["P1_立项依据"],
      "citation_positions": ["P1, 第2段, 第3句"],
      "role": "gap_evidence",
      "key_finding": "发现XXX通过YYY通路调控ZZZ",
      "search_source": "pubmed",
      "verified": false,
      "verification_details": {
        "title_match": null,
        "doi_valid": null,
        "pmid_match": null,
        "retracted": null,
        "checked_at": null
      }
    }
  ]
}
```

**role 枚举值：**
- `gap_evidence`：证明研究空白/未解决问题
- `method_support`：支撑技术路线/方法选择
- `prior_work`：申请人前期工作
- `comparison`：对比参照/不同观点
- `background`：领域背景知识

## 四、文献验证流程（强制，零容忍）

### 4.1 验证时机

每次新增文献到 literature_index.json 后，**必须立即执行验证**，不得延迟到后续Phase。

### 4.2 验证步骤

```
对每条新增文献执行：

Check 1: 标题真实性
  - 用DOI或PMID在PubMed/Semantic Scholar反查
  - 确认返回的标题与录入标题一致
  - 不一致 → 标记 title_match: false → 修正或删除

Check 2: DOI有效性
  - 确认DOI格式正确且可解析
  - 无效 → 标记 doi_valid: false → 补充或修正

Check 3: PMID对应性
  - 确认PMID对应的论文与录入信息一致
  - 不对应 → 标记 pmid_match: false → 修正

Check 4: 撤稿检查
  - 检查论文是否已被撤稿
  - 已撤稿 → 标记 retracted: true → 删除并替换

Check 5: 正文引用核对
  - 确认P1中引用该文献的上下文与key_finding一致
  - 不一致 → 修正引用上下文或key_finding
```

### 4.3 验证状态管理

```
verification_status 枚举：
  "pending"   — 未验证
  "verified"  — 全部通过
  "partial"   — 部分通过，有警告
  "failed"    — 验证失败，需修正
```

**门控规则：** Phase 1完成时，所有文献的 `verified` 必须为 `true`，否则阻断进入Phase 2。

### 4.4 引用-索引一致性矩阵

每次修改P1或REF后，自动生成并检查：

```
引用矩阵检查项：
  1. P1正文中的每个[N]引用 → literature_index中必须有ref_number=N的条目
  2. literature_index中每个条目 → 如used_in_sections包含P1，则P1中必须有对应[N]引用
  3. REF_参考文献.md中的条目顺序 → 必须与P1中首次引用顺序一致
  4. 无孤立引用（正文有[N]但索引无对应）
  5. 无孤立条目（索引有条目但正文未引用）
```

矩阵检查失败 → 输出具体不一致项 → 修正后重新检查。


## 五、双轨校验策略（MCP + 官方来源）

为降低误检率，文献核验采用双轨机制：

1. 轨道A（召回轨）：通过 `paper-search` MCP 获得候选条目，缓存到 `data/mcp_literature_cache.json`。
2. 轨道B（核验轨）：通过 PubMed/Crossref 官方接口做 DOI/PMID/标题反查与撤稿状态校验。
3. 两轨交叉：若 MCP 与官方来源冲突，标记失败并输出失败原因（如 `title_mismatch`、`id_mismatch`、`retracted`、`source_unreachable`）。
4. 运行时统计：每次核验写入 `data/verification_run_log.json`，记录耗时、失败类型分布、平均置信度。
5. 人工复核分流：冲突条目自动写入 `data/manual_review_queue.json`，供人工逐条裁决。

`mcp_literature_cache.json` 推荐结构：

```json
{
  "entries": [
    {
      "title": "...",
      "doi": "10.xxxx/...",
      "pmid": "12345678",
      "retracted": false
    }
  ]
}
```

也支持键值格式（按 DOI/PMID 作为键）：

```json
{
  "doi:10.xxxx/yyy": {
    "title": "...",
    "doi": "10.xxxx/yyy",
    "pmid": "12345678",
    "verified_at": "2026-02-27"
  }
}
```

执行建议：
- 联网时：优先执行双轨（不要加 `--offline`）。
- 网络受限时：允许 `--offline`，但该状态仅作临时通过，后续必须补线上核验。
- 建议开启 `--mcp-ttl-days 30`，超过 TTL 的 MCP 缓存会标记 `mcp_stale` 并进入人工复核队列。
