# 06 发布

目标：建 GitHub 仓库、写能让陌生人上手的 README、走 PR + CI 合并发布。

## Step 1 ⏸ 卡点4：发布信息确认（先问再动）

问用户：

1. **仓库名**（给个建议名）
2. **公开还是私有**（提醒：公开仓库 GitHub Actions 免费）
3. **主分支名**（默认 main）
4. README 里有没有要补充的内容（徽章、致谢、License 等）
5. 发布形态：只推代码 / GitHub Pages / Release + tag / npm 等

这一个卡点覆盖后续所有 push 和公开动作，中途不再逐次问。

**反向拷问（发布是不可逆的公开动作，值得挑一次）**：确认发布信息前，若用户默认"该发了/公开"，给一条【反向拷问】冲着"是不是真做完了、该不该现在公开"去——挑没验证的完成假设（核心流程真跑通了吗、公开vs私有的默认对不对、有没有为赶进度跳过的东西）。≤100字，一条，挑完等用户拍板。用户已明确权衡过公开范围和完成度的，不复读。

## Step 2：建仓库

```bash
gh repo create <名字> --private|--public --source=. --remote=origin
```

没装 gh 或没登录：把命令给用户自己跑，或引导 `gh auth login`。

## Step 3：README（必须详细，写给没看过代码的人）

必含章节：这是什么、能干什么（功能清单）、**安装步骤**（从零开始，含依赖）、**部署步骤**（如适用）、**使用教程**（带真实示例命令/截图位）、常见问题。写完自查：一个没参与开发的人照着 README 能不能跑起来。

## Step 4：GitHub Actions

按项目类型生成最小 workflow（`.github/workflows/ci.yml`）：装依赖 → lint/typecheck（项目有就跑）→ 跑单测 + 验收测试 → 依赖漏洞扫描（Node 用 `npm audit --audit-level=high`，Python 用 `pip-audit`）。网页项目**必须**在跑测试前加 `npx playwright install --with-deps`（CI 里没浏览器，缺这步验收测试必红），酌情加构建检查。**BRIEF 里有多平台要求的，用矩阵跑对应系统（`runs-on: [ubuntu-latest, windows-latest, macos-latest]` 按需选）——05 标"待 CI"的跨平台用例在这里判定，红了照样回 04 修。**不堆用不上的花活。

## Step 5：PR → 合并 → 发布

1. **推送前密钥终检**（一票否决，尤其公开仓库）：`.env` 等敏感文件确实被 `.gitignore` 排除、`git log -p` 抽查历史里没混进过 key、源码无硬编码凭证——05.5 审过，这里再花一分钟终检，push 出去就收不回了
2. push feature 分支，建 PR（PR 描述写清做了什么、测了什么）
3. 等 Actions 绿——红了回 04 修，**不许绕过 CI 强行合并**
4. 合并进主分支
5. 按 Step 1 确认的形态发布：Pages 开启 / `gh release create` + tag / npm publish 等

## 完成标准

用户在卡点4确认的所有发布动作完成、CI 绿、README 在线可读 → 进入 07 收尾。
