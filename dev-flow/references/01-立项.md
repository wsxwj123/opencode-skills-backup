# 01 立项

目标：把一句模糊的想法变成一份没有歧义的需求定稿（BRIEF.md）。

## Step 0：确认工作文件夹（第一件事，先于一切）

问用户：**这个项目做在哪？**
- 就在当前会话所在文件夹，还是
- 新建一个项目文件夹（问清路径和文件夹名）

路径确认前不许创建任何文件。

## Step 1：grilling 拷问到极致

调用 `grilling` 技能（Skill 工具，skill: "grilling"），对需求逐个问题拷问，覆盖：

- **动机**：为什么要做这个？解决什么真实问题？已有现成工具为什么不用？
- **用户和场景**：谁用？在什么场景下用？多久用一次？
- **核心功能**：最小可用版本必须有什么？每个功能能不能说成"输入 X 应该得到 Y"？
- **边界**：明确不做什么？（不写下来的边界日后都会变成扯皮）
- **成功标准**：做到什么程度算做完？怎么验证？
- **项目类型**：代码/脚本、网页/前端、skill/prompt，还是混合？
- **平台范围**：只跑 Mac？要兼容 Windows/Linux？网页要兼容哪些浏览器？——这直接决定测试策略
- **技术偏好**：有指定语言/框架/库吗？没有就留给方案阶段调研决定
- **数据与依赖**：需要外部 API、数据库、登录态吗？
- **敏感面（安全审计的依据，三问必答）**：①程序会碰哪些用户文件/数据（读、写、删的范围各是什么）？②要不要密钥/登录态，存哪？③会往哪些外部地址发数据？——答案写死进 BRIEF，05.5 安全审计按它核对"程序有没有越权"
- **发布意图**：要不要上 GitHub？公开还是私有？要不要部署上线？

**拷问停止标准**：每个功能都能写成"输入 X 应得到 Y"，且边界、成功标准、平台范围全部落字。有一处模糊就继续问。

## Step 1.5：反向拷问（grilling 之后，仅当用户带着现成方案/强主张来）

grilling 把"要什么"问清后，如果用户不只是提了个问题、而是**自带解决方案或强主张**（"做个X用Y""必须自己写不用现成的""一定要上线"），进方案阶段前给一条【反向拷问】：挑他没验证的前提、忽略的盲点、或更短的路径（现成工具能不能直接解决、这需求是不是伪需求、最小版本能不能再砍）。≤100字，只挑最要害一条，挑完等回应；用户坚持就照走、把风险记进 BRIEF。用户只是来问"能不能做/怎么做"、还没定方案的，跳过这步。

## Step 2：定测试策略

根据项目类型和平台范围，写明测试怎么做（进 BRIEF.md）：

| 类型 | 验收测试形态 |
|---|---|
| 代码/脚本 | pytest / node test 调用对外接口（CLI 命令、函数），真实跑 |
| 网页/前端 | 起本地服务 + 浏览器自动化走真实用户流程 |
| skill/prompt | 测试 prompt 清单 + 产出评分标准，裁判盲测实际产出 |
| 多平台要求 | 立项时定哪些平台必测、哪些平台"尽力而为"，写死进 BRIEF |

## Step 3：初始化

1. `git init` + 合适的 `.gitignore`
2. 创建 `.devflow/` 目录，写入 `BRIEF.md`：背景与动机 / 功能清单（每条"输入X→得到Y"）/ 明确不做的事 / 成功标准 / 项目类型与平台范围 / 测试策略 / **敏感面声明** / 发布意图
3. 在项目根创建 `PROJECT.md` 状态账本（格式见 04-并行worktree.md 末节；唯一写者=主会话），登记初始待办
4. 首次 commit（message：`chore: 立项，需求定稿`）

## 完成标准

BRIEF.md 落盘且用户对拷问结论无异议 → 进入 02 方案。
