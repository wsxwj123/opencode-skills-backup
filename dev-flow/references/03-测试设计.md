# 03 测试设计

目标：在写第一行实现代码**之前**，由独立代理站在用户角度写好验收测试并锁定。

为什么这么严：让 AI 看着自己写的代码出测试，等于学生自己出考卷自己判卷——测试会不自觉迁就实现，代码错了测试跟着错。调研证据（AgentCoder、SpecBench 等）一边倒：写测试的不许看实现。

## Step 1：测试设计代理（opus，独立子代理）

**给它的输入只有**：`.devflow/BRIEF.md` + `.devflow/INTERFACE.md`（02 单独抽出的接口约定）。**不给 PLAN.md**——里面的文件结构和任务拆解是实现思路，看了黑盒就破功。

**Prompt 里明确写**："不许读取项目里任何实现代码（现在还没有实现，之后也轮不到你看）。只依据需求和接口约定，站在真实用户的角度设计测试。"

要求覆盖（缺哪类要说明为什么不适用）：

1. **正常路径**：BRIEF 里每条"输入X→得到Y"的主路径，典型输入→预期输出。
2. **边界**：0、1、空、负、极大极小、临界值、单元素、超长、Unicode/特殊字符。
3. **错误路径（最易漏，重点）**：乱输入、空输入、缺参、类型错、重复操作、误操作、文件缺失、断网、越权、依赖不可用——必须被正确拒绝或兜底，不能崩、不能静默出错；每种断言"具体的"失败行为（确切的退出码/异常类型/报错文案），不是笼统"会报错"。
4. **反向用例**：不该发生的事断言它没发生（不该写库的没写、不该返回的没返回、**不该删的文件还在**）。
5. **幂等/并发**（若适用）：重复调用、并发调用、重复运行不产生脏数据。
6. **用户没想到的也要想到**：需求没明说但按常理必须正确的行为（中文文件名、路径带空格等），按 BRIEF 定的平台范围顾兼容性。

每个测试只测一件事，命名说清场景；断言要具体（确切值/异常/退出码，别只断言非空/不报错）；测试间不许有依赖、能任意顺序跑；不引入没必要的框架和 fixture。交付时明说覆盖了什么、没覆盖什么，不假装全覆盖。

## Step 2：产出（按项目类型分叉）

| 类型 | tests/acceptance/ 里放什么 |
|---|---|
| 代码/脚本 | 可执行测试（pytest / node test），只通过对外接口调用（CLI、公开函数），不 import 内部模块 |
| 网页/前端 | **`@playwright/test` 的 spec 测试文件**（npm 标准测试框架，带断言/报告，`npx playwright test` 可复跑、可进 CI）走真实用户流程：打开页面→操作→断言看到的结果。注意：playwright **MCP** 和 agent-browser 是交互式驱动工具，产不出可提交复跑的测试文件，**不能当验收测试产物**（它们的定位是 04 开发期探路调试） |
| skill/prompt | 测试 prompt 清单 + 每条的产出评分标准（什么样算合格），供 05 裁判盲测 |

同时产出 `.devflow/TEST-PLAN.md`（人话版清单）：每条一行——场景 / 怎么操作 / 预期看到什么。这是给用户看的。

## Step 3：守卫检查（主会话执行）

检查验收测试内容：**不得出现 INTERFACE.md 之外的实现细节**（内部函数名、私有模块、数据库表名等）。出现 = 黑盒破功，打回 Step 1 重写。

## Step 4 ⏸ 卡点2：清单锁定

把 TEST-PLAN.md 给用户过目（说人话）：一共多少条、覆盖哪些方向、哪几条是"替你想到的"。用户可增删。

确认后锁定——hash 必须落盘（05 从文件读，不靠会话记忆，防长会话遗忘）：

```bash
git add tests/acceptance .devflow/TEST-PLAN.md && git commit -m "test: 验收测试锁定"
git rev-parse HEAD > .devflow/LOCK
git add .devflow/LOCK && git commit -m "chore: 记录验收测试锁定 hash"
```

此后开发代理对 `tests/acceptance/` 只读。

## 修 bug 模式

核心产物是**复现测试**：修之前跑必须失败（证明真复现了 bug），修之后必须通过。另加 1-2 条相邻场景的回归用例（防止修好一处坏了旁边）。同样锁定后进 04。
