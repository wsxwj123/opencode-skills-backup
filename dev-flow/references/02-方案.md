# 02 方案

目标：产出一份经过独立盲审的方案定稿 PLAN.md。出方案和挑毛病必须是两个互不知情的子代理。

输入：`.devflow/BRIEF.md`（没有就先按 01 补一份轻量版，至少把功能清单和成功标准落字）。

## Step 0：已有项目加功能时，先补一轮需求拷问（新项目跳过——01 已拷过）

从 02 进入（给已有项目加功能/改需求）时，别直接调研——**先针对这个新功能做一轮 grilling 拷问**（同 01 Step1，但只聚焦本次新增/变更）：这个功能要解决什么、输入什么得到什么、边界在哪、成功标准、会不会碰新的数据/权限。用户带着明确做法来的，再补一条【反向拷问】。把结论并进/更新 BRIEF.md，再往下走。避免"加功能不问清就闷头做"。

## Step 1：调研（有需要才做，opus 子代理）

先判断方案需要哪些调研：技术选型对比、官方文档用法、有没有现成轮子可复用、平台限制等。

- 每个调研主题派一个**独立 opus 调研子代理**，可并行
- 要求：优先查官方文档和源头资料；产出 `.devflow/RESEARCH-<主题>.md`，含结论、依据、来源链接
- 生态里可能有现成 skill 能用时，调用 `find-skills` 技能查询（它自带查询命令说明），装不装问用户
- 方案必须基于调研写，不许凭记忆猜 API 用法

## Step 2：方案代理（opus，独立子代理）

给它的输入：BRIEF.md 路径 + 所有 RESEARCH 文件路径 + rules.md 路径。

要求产出 `.devflow/PLAN.md`，必须包含：

1. **架构与技术选型**（附选型理由，引用调研结论）
2. **文件结构**
3. **对外接口约定**：CLI 命令及参数 / 页面 URL 与操作入口 / 公开函数签名 / skill 的调用方式——写到"不看实现代码也能照着写测试"的程度，**且必须写明每个入口的错误契约**（非法输入返回什么：退出码、异常类型、报错文案——否则测试设计的反向用例只能瞎猜预期）。**单独存一份 `.devflow/INTERFACE.md`**（只含这一节）。测试设计代理只会拿到 INTERFACE.md，不给完整 PLAN——物理隔离，防止实现思路泄露给测试设计。
4. **任务拆解**（按模块，标注顺序依赖；互不依赖的模块标注"可并行"，供 04 按 04-并行worktree.md 判定）
5. **风险清单**（按 rules.md 风险评估规则：每个非机械改动至少 1 个失败模式+缓解；**BRIEF 敏感面声明的每一项，配一句"怎么保证不越权/不泄露"**）

## Step 3：审查代理（opus，独立子代理，盲审）

**Prompt 必须这样开头**："这是隔壁同事做的方案，你详细审查一下。" 绝不透露方案出自另一个 AI 或本会话。**并加白名单式禁令**："只依据我给你的这两个文件（BRIEF.md、PLAN.md）判断；除这两个文件外，不得查看 .devflow/ 下任何其他产物、不得查看 git 历史、不得查看实现代码。"

给它的输入：BRIEF.md + PLAN.md 路径。要求：

- 按 致命 / 重要 / 建议 三级列出问题
- 重点挑：需求覆盖缺漏、架构隔离问题、接口约定含糊处、被低估的风险、过度设计
- **必查一项**：INTERFACE.md 的错误契约是否完整（每个入口对非法输入返回什么都写清了吗）——缺了，03 的反向用例就只能瞎猜，判为"重要"问题打回补全
- **四维度审查清单（逐个过，不许跳；只查方案层，不查代码——现在还没有代码）**：
  1. 正确性：接口约定的错误契约是否完整、边界场景（空/0/超长/并发/重复操作）方案有没有想到、状态与数据流设计有没有漏态。
  2. 安全：BRIEF 敏感面声明的每一项方案里有没有对应保障、信任边界在哪、有没有注入面/密钥暴露的设计隐患。
  3. 性能：架构层面的明显低效设计（如方案就注定 N+1、全量扫描），不猜实现细节。
  4. 可维护性：过度设计（单实现的抽象、用不上的配置、为想象需求预留的扩展点，一律点出来砍）、模块拆分是否合理。
  基调是帮方案改好不是显摆；每条问题给"位置+为什么是问题+怎么改"；优先级排序，先说最要命的。
- **架构重的改动可选**：若本次改动架构层面重，可再派一个架构审查代理（同样盲审、白名单），专挑模块边界与职责、依赖方向与循环依赖、设计模式是否硬套、可演进性（加功能会不会散弹式修改）。
- 明确说"没发现致命问题"也算合格产出，不许为了交差硬凑

## Step 4：修订

把审查意见回传方案代理修订（致命/重要必须处理，建议可说明不采纳理由）。重大分歧再走一轮 Step 3。

## Step 5 ⏸ 卡点1：方案定稿

向用户汇报（说人话）：方案要做什么、怎么做、审查挑出了什么毛病、怎么改的、还剩什么风险。

**必须让用户看到并确认方案，按当前模式选方式**：
- **规划/plan 模式下**（Claude Code plan、Codex、OpenCode 等 agent 的规划态）：调用对应 agent 的计划卡片工具展示方案（Claude Code 用 ExitPlanMode 出计划卡片；其他 agent 用其对应的计划确认工具），让用户在卡片上拍板。
- **编辑/放任/其他非规划模式下**：提醒用户去查看 `.devflow/PLAN.md` 文件（给出路径），别只在对话里口头概括就往下走——方案是要落字确认的东西。

**反向拷问（仅当用户在卡点1推翻审查意见、或坚持一个审查没覆盖的选择时触发）**：审查代理已经挑过方案的技术缺陷，这里**不重复挑技术**。只在**用户自己的决定**需要挑时开口——用户否决了审查的关键意见、或坚持某个约束/选型时，给一条【反向拷问】挑这个坚持背后的假设（≤100字）。用户被动确认方案、没额外主张的，不拷问，直接定稿。

用户确认 → PLAN.md 定稿，commit（`docs: 方案定稿`）→ 进入 03。

## 修 bug 模式（轻量版）

1. **理解与复现**：先真实复现 bug，确认触发条件。
2. **找根因**：grep 所有相关调用方，追到根因，不打症状补丁。修复点应在所有调用方共同经过的位置。
3. 写简版 `.devflow/PLAN.md`：bug 现象 / 根因 / 修法 / 影响面（哪些文件、哪些调用方）。
4. ⏸ 卡点1：根因和修法向用户说清（说人话），确认后进入 03——修 bug 模式下 03 的核心产物是**复现测试**（修之前跑必须失败，修之后必须通过）。
5. 修完验收后**不必走完整 06**：已有仓库的小修，直接 commit + push（或小 PR），跳过 README/建仓/Release 那些步骤。
