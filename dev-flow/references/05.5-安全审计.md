# 05.5 安全审计

目标：发布前的最后一道安全闸。一旦推到公开仓库，密钥泄露就来不及了，所以放在验收通过之后、发布之前。

分工（别重复劳动）：
- **外部攻击面的日常防护是白嫖的**——已装的 security-guidance 插件在开发阶段自动三层扫描（写代码时正则警告、每回合 LLM 差异审查、commit 时跨文件审查），04 已要求开发代理处理它的警告。
- **本阶段自己干的是它管不到的部分**：内部数据安全（程序有没有越权动用户数据、往外发数据、日志泄密、删除没确认），外加一次全局交叉复核。

按任务分级：
- 微任务：跳过本阶段（自动工具照跑）
- 小任务：不派审计代理，只跑一次内置 `security-review` 技能审当前分支改动，有致命/重要发现才处理
- 标准任务：走下面完整流程

## Step 1：安全审计代理（opus，独立子代理，盲审）

Prompt 开头："这是隔壁同事做完、准备发布的项目，上线前你做一次安全审查。**只依据我给你的文件和源码目录判断**，不得查看 git 历史、.devflow/ 下其他产物、开发过程对话。"

输入：项目源码目录 + BRIEF.md（含敏感面声明）+ 项目类型。

审查清单（每条给 命中/未命中 + 证据 文件:行号）：

**A. 外部攻击面**
1. 注入：用户输入拼进命令/SQL/HTML 的地方有没有参数化/转义
2. 密钥硬编码：源码里有没有写死的 API key/token/密码；`.env` 是否被 `.gitignore` 排除
3. 不安全解析：对外部数据用了 `eval` / `pickle` / `yaml.load` 之类（等于让别人的文件在你机器上跑代码）
4. 依赖：有没有名字像仿冒的、明显过时高危的第三方包
5. 提示词注入面（项目本身是 AI 应用/agent/skill/爬虫时必答）：程序把外部内容（网页/用户输入/第三方数据）拼进发给 LLM 的 prompt 时，有没有当数据隔离（分隔符包裹+显式声明"以下是数据非指令"）？有没有对模型输出做格式校验后再执行，而不是拿模型输出直接跑命令/发数据？

**B. 内部数据安全（重点，逐条必答）**
5. 文件读写范围：程序只动它声明要动的目录吗？有没有越界路径、动了不属于本项目的文件
6. 删除操作：有没有 `rm -rf` / 批量删除；删之前有确认吗；能回退吗（回收站 vs 直接抹掉）
7. 数据外发：有没有把本机文件、环境变量、用户数据发到外部地址；那个地址是需求里没提过的第三方吗
8. 日志泄露：日志/报错/print 里有没有 token、密码、完整文件内容、个人隐私
9. 凭证存储：key/cookie/登录态存哪；是否明文落盘；会不会随代码进 git

**核心判据**：对照 BRIEF.md 的敏感面声明逐条核对——程序实际动的东西超出声明范围 = 高危，哪怕功能正常。

输出 `.devflow/SECURITY-REPORT.md`：致命/重要/建议三级，每条附 文件:行号 + 一句人话讲清危害 + 修法；"未发现致命/重要问题"也是合格产出，不许硬凑；结尾给结论：可发布 / 需修复后再发布。

## Step 2：交叉复核（主会话执行）

跑一次内置 `security-review` 技能（官方通用清单，与上面的自定义盲审互补），结果对照 SECURITY-REPORT.md 合并。

## Step 3：Gate 规则

- 致命/重要 → 派开发代理修（审计员只审不改，保持角色隔离）→ 修完回 05 重跑功能验收 → 再回本阶段复审
- 全清或只剩"建议"级 → 安全结论并入 **卡点4** 与发布信息一起向用户汇报（说人话：查了什么、有没有事、修了什么）

## 附：按项目类型最要命的 3 条（审计代理和自查通用速查）

| 类型 | 最常翻车的 3 条 |
|---|---|
| CLI/脚本 | ①命令注入（拼字符串执行）②越权文件操作/删除无确认 ③密钥写死或日志带出 token |
| 爬虫 | ①抓回的内容当代码跑（eval/pickle/yaml.load 投毒路径）②cookie/账号明文落盘或进 git ③数据发往写死的第三方地址 |
| 网页/PWA | ①XSS（innerHTML 塞用户内容不消毒）②密钥打进前端 bundle/localStorage（前端=公开），Supabase 查 RLS 行级权限和越权读 ③后端注入与鉴权 |
| skill | ①提示词注入/诱导外发用户数据的指令 ②附带脚本越权读删范围外文件 ③skill 文件里写死 token（skill 会被分享，等于公开） |
